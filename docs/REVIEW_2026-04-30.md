# MDK Concept Review
**Date:** 2026-04-30  
**Reviewer:** Claude Sonnet 4.6  
**Scope:** Follow-up to 2026-04-27 review. Covers the design session that produced `examples/construct-pattern/`, the CDK/CDKd interoperability discussion, the provider model, NEAT calibration scope, Giannantoni IDC, and the building-exhibition conversation about commercial opportunity.

---

## What Changed Since the Last Review

The 2026-04-27 review said: cut to three phases, stop adding to the roadmap, the target user is too small.

This session did not cut — it expanded. But it expanded in the right direction and for good reasons. The conversation clarified something that was previously implicit: MDK is not primarily a mechatronics tool. It is a general systems simulation and infrastructure framework whose first concrete domain happens to be mechatronics/ecology/hydrology. That is a larger claim and a more defensible one. The question is whether the execution can stay focused enough to prove the claim on one domain before fragmenting across all of them.

## Ideas Evaluated

### 1. CDK-style L3 Constructs (`ModelConstruct` / `ModelStack`) — **Strong. Build this.**

The `examples/construct-pattern/` directory sketched what the API should look like. The key insight, arrived at by discussing AWS CDK's critics, was: **Construct = pure physics. Stack = synthesis boundary. Never mix them.** `EcosystemStack` instantiates `HydrologicalSystem` and `SensorNode`; it does not define any nodes or edges itself. This is the right discipline and the main thing missing from the existing `MdkSystem`/`MdkStack`/`MdkApp` API.

The existing low-level API stays. The L3 CDK-style layer sits on top of it. No breaking change.

**Verdict:** This is the next major implementation task. `ModelConstruct` and `ModelStack` as proper CDK-pattern base classes, with `Store`/`Source`/`Sink`/`Flow` as Odum ESL L3 constructs. Add to TASKS.md as T2.6 and T2.7, EXT scope.

---

### 2. Site Registry Pattern — **Strong. Clarifies the 100-site problem.**

The question "what if there are 100 sites, do we write 100 TypeScript files?" has a clear answer: no. `SiteRegistry.load('./sites.db')` returns `SiteConfig[]`; you iterate once; one `EcosystemStack` class instantiates once per site. This is exactly the CDK multi-account/multi-region pattern. It works for Hydstra sites, industrial sensor fleets, ecological monitoring networks, any cardinality.

The `Site` construct (anchors to lat/lon, carries Hydstra data sources, owns the CBOR output inference) is the right abstraction. CBOR output should be inferred from Site ancestry, not declared per-node.

**Verdict:** `Site` construct and `SiteRegistry` are part of `@mdk/domain-odum` (or `@mdk/provider-odum`, see item 5). Not `@mdk/core`. Core stays domain-agnostic.

---

### 3. BG/Odum Validation Mapping Table — **Strong. Genuinely novel observation.**

The table mapping BG structural audit rules to their Odum ESL equivalents (see USERS_MANUAL.md section 9, added in this session) is not documented anywhere else. It is the theoretical justification for using a single validation engine for both domains. The validator becomes a parameterised graph rule engine: BG rules are the canonical form; each domain registers a dialect that either adds rules or translates to BG form.

TypeScript class inheritance (`OdumConstruct extends ModelConstruct`) is the right mechanism for construct-level composition. But the validation rule inheritance is a separate concern: a `ValidationRuleSet` registry where each domain declares `{ inherits: 'bondgraph', additional: [...] }`. These are two orthogonal extension axes.

**Verdict:** Document this in USERS_MANUAL.md (done). Implement `ValidationRuleSet` registry as part of T2.6.

---

### 4. AWS CDK + MDK Interoperability — **Correct but the framing evolved.**

Initial position: "CDK infra constructs live in a CDK Stack, not an MDK ModelStack — keep them separate." The user correctly pushed back: the CDKd concept breaks that orthodoxy. The simulation model should be the specification for infrastructure AND the test oracle for the running system.

The revised framing: `app.synth()` should produce multiple artefact types — CloudFormation template, simulation JSON, and a validator Lambda that periodically queries CloudWatch/IoT metrics and compares against the simulation prediction. "How it started" (sim output) vs "how it is going" (CloudWatch actuals) is the CDKd feedback loop.

This is architecturally possible because both MDK's `ModelConstruct` and `aws-cdk-lib`'s constructs share the same `constructs` npm base class. They can coexist in the same scope tree with no adapter.

**Verdict:** Pattern A (side-by-side stacks) is right for production deployments. Pattern B (single stack with both MDK + CDK constructs) is right for tight integrations and prototypes. The CDKd loop (sim-as-specification → deploy → validate → compare) is the most commercially differentiated idea in this entire project. It deserves a dedicated task: T4.4 CDKd Validation Lambda. EXT scope.

---

### 5. Provider Model vs Domain Splitting — **User is right. Don't split.**

The suggestion to have `@mdk/domain-odum` and `@mdk/domain-ecology` as separate packages is wrong. Lotka-Volterra equations are Odum ESL Interaction flows. Nutrient cycling is Odum ESL. Predator-prey is Odum ESL. There is no separate ecology domain — it is all the same mathematics with different semantic labels.

The right model is Terraform providers, not CDK domains:
- `@mdk/core` — BG primitives, validation engine, WASM bridge, L1/L2 constructs
- `@mdk/provider-odum` — Odum ESL L3 constructs (`Store`, `Source`, `Sink`, `Flow`, `Site`), Odum validation rules, Hydstra integration
- `@mdk/provider-aws` — AWS CDK integration constructs, IoT fleet provisioning, S3 CBOR store
- `@mdk/provider-maxon`, `@mdk/provider-raspberrypi`, etc. — physical component catalogs

**Verdict:** Rename all `@mdk/domain-*` references to `@mdk/provider-*` throughout the codebase and docs. Update TASKS.md accordingly.

### 6. NEAT + Parameter Calibration in Scope (C/WASM) — **Yes, put it in scope.**

The user asked why NEAT is not in scope, and the reasoning is sound: if you are ingesting a real Hydstra timeseries and the simulation does not match, there are exactly two explanations:
1. The parameters (`k` values, initial conditions) are wrong → PINN / gradient-free calibration fixes this
2. The model structure (wrong number or type of nodes/edges) is wrong → NEAT fixes this

Both are needed for a complete calibration story. Neither should be in TypeScript — the inner loop (fitness evaluation) is already WASM. The population management in NEAT operates on JSON model specs and can be TypeScript; the kernel evaluation is WASM.

Implementation note: `advanced.c` already has ensemble forecasting. The calibration harness (parameter optimization via Nelder-Mead or CMA-ES) extends `advanced.c`. NEAT population management in TypeScript mutates the JSON model spec; each candidate is evaluated by calling `runKernel()`.

**Verdict:** Add T0.8 (parameter calibration harness — CMA-ES extending `advanced.c`) and T0.9 (NEAT structure discovery — TypeScript population manager + WASM kernel evaluator) to TASKS.md as EXT scope. These are not MVP but they are no longer Research — the path is clear.

### 7. Giannantoni IDC / Emergy Integration — **Scope clarification needed.**

T0.6 in TASKS.md is correct — IDC is currently Research scope requiring a mathematics partner. The user's question "where is the Giannantoni IDE incipient stuff?" reveals the motivation more clearly than the task description does.

The practical case for IDC in MDK is not just mathematical elegance. It is specifically:
- **Continuous emergy accounting**: IDC gives exact emergy trajectories for the ODE classes it handles (Riccati, linear variable-coefficient ODEs up to order 4). TDC-based emergy accounting has discretization error that accumulates over long simulations. For ecological models that run for years, this matters.
- **Self-organising systems**: Giannantoni's claim that IDC captures generative causality (not just efficient causality) is contested in mainstream physics but is directly relevant to Odum's Maximum Em-Power Principle. Running both solvers and exposing drift empirically is exactly the right scientific stance.

IDC is not implementable without a CAS for the general case. But for the specific Odum ESL ODE classes that Giannantoni solved explicitly (Riccati = interaction/feedback models, linear variable-coefficient = time-varying inflows), the closed-form solutions can be implemented directly in C. That subset is genuinely useful for ecological simulation.

**Verdict:** Keep T0.6 as Research but add the specific Odum ESL application (continuous emergy tracking via IDC closed forms) as the concrete motivation. Move the Riccati implementation subitem from Research to EXT — it is tractable without a CAS partner.

### 8. Component Catalog with Commercial Metadata — **Strong commercial differentiator.**

The existing `mdk-package.json` schema already carries BG parameters from datasheets. The natural extension is a `commercial` block:

```json
{
  "commercial": {
    "distributors": [
      { "name": "Maxon Direct", "sku": "386866", "priceBreaks": [{ "qty": 1, "priceUSD": 285 }] }
    ],
    "leadTimeDays": 14,
    "minOrderQty": 1,
    "tolerance": { "R_a": 0.10, "L_a": 0.20 }
  }
}
```

Then `mdk synth --bom` traverses the construct tree, collects all L3 component constructs, and outputs a Bill of Materials with prices, lead times, and tolerance ranges. Monte Carlo simulation (T8.2) consumes the tolerance ranges to produce uncertainty bounds on simulation outputs.

This is the thing that would make an HVAC engineer or a field ecologist care about MDK. Not the Bond Graph theory — the ability to go from "I described my system in code" to "here is what it costs and when I can get it."

**Verdict:** Extend the `mdk-package.json` schema with a `commercial` block. Add `mdk synth --bom` as a new output target. EXT scope, medium priority.

## Honest Problems That Remain

### The user base is still small

The 2026-04-27 review said the intersection of "TypeScript engineers" + "Bond Graph knowledge" + "MBSE practitioners" is tiny. This session expanded the target user by adding: ecologists doing timeseries modelling, water resource engineers using Hydstra, building services engineers at trade exhibitions. These are better targets — they don't need to know Bond Graph theory, they just need to use Odum ESL constructs.

But the onboarding problem remains unsolved. The `examples/construct-pattern/` files are the best documentation so far. A GitHub Pages interactive demo (T5.4) remains the highest-priority unbuilt thing for user acquisition.

### The CDKd loop is compelling but complex

The "simulation-as-specification → deploy → validate → compare" loop is the most commercially interesting idea here. But it requires:
1. A stable simulation output schema (done — CBOR timeseries)
2. A CloudWatch/IoT metric schema that maps to simulation state variables (new work)
3. A validator Lambda that runs on a schedule and computes divergence (new work)
4. An alerting/reporting layer that makes the divergence actionable (new work)

None of this is blocked by physics. It is all software engineering. But it is a substantial implementation surface that should be sequenced carefully: get the simulation output stable first, then build the validator.

### The provider model needs a cold-start strategy

`@mdk/provider-odum` is most of the value for ecological/hydrological users, but it does not exist yet as an implemented package — only as design sketches. Until it is implemented with real Hydstra integration, the Site/SiteRegistry pattern is fiction. Implementing `@mdk/provider-odum` as a real package with a working `Site.fromHydstraApi()` is the concrete next step.

## Blue Sky: Where Should MDK Go

The following are not near-term tasks. They are directions. Some are further-fetched than others.

### Physical domain providers

**Building and construction** — the Sydney building exhibition observation is real. Every discipline in commercial construction needs systems integration: HVAC, building envelope, structural, fire safety, acoustic, hydraulics, electrical. Each discipline has its own simulation tools (EnergyPlus for thermal, ETABS for structural, etc.) and none of them talk to each other. An MDK `@mdk/provider-hvac` with Daikin, Carrier, or Trane component constructs could produce a cross-discipline energy balance model that no current tool generates.

**Hydraulics and water** — `@mdk/provider-hydraulics` for pipe networks, open-channel flow, weir hydraulics, pump curves. Directly adjacent to the Hydstra/water resource work. The same Bond Graph C/R/I framework handles pipe flow (inertance, friction, compliance of pipe walls) natively.

**Power systems** — microgrids, battery storage, solar + grid integration. The electrical Bond Graph domain already handles this. `@mdk/provider-energy` with typical solar panel, inverter, and battery constructs would be immediately useful for off-grid IoT deployments (the sensor node example) and for building energy management.

**Pharmacokinetics and drug delivery** — compartmental models (identical mathematics to Odum ESL ecological compartments) are the standard in pharmacology for modelling drug distribution in the body. `@mdk/provider-pharmaco` would make MDK relevant to drug development. The Interaction flow type models drug-receptor binding; the Monod/Limit type models enzyme saturation. A long-shot but technically zero new work in the kernel.

**Chemical process engineering** — HAZOP (Hazard and Operability Analysis) and P&ID (Piping and Instrumentation Diagram) as MDK constructs. The process industry has legally mandated risk analysis workflows that are currently done in Word documents and Excel spreadsheets. An MDK `@mdk/provider-process` that generates P&ID diagrams from the model and feeds into HAZOP templates would be high value in oil/gas, pharma manufacturing, water treatment.

**Financial and economic systems** — Odum ESL was applied to economic modelling by Odum himself (energy economics). Lotka-Volterra dynamics model market competition; compartmental models describe supply chains and inventory flows. `@mdk/provider-economics` is speculative but theoretically grounded. Investors in quantitative finance pay serious money for simulation tools.

### CAD integration

If L3 component constructs have physical dimensions and mass properties (not just BG parameters), `mdk synth --cad` could export a STEP assembly file. This means the MDK model IS the CAD model at the system level — not a separate design tool, not a parallel document. Every component's position in the assembly is derived from the construct tree hierarchy.

The dependency: L3 constructs would need a `geometry` block in `mdk-package.json` (bounding box, mounting features, mass, CG). For standard components (motors, batteries, PCBs), this is available from datasheets. For custom parts, the user provides it. The export format is STEP AP214, which every CAD tool can read.

This would make MDK genuinely competitive with SysML IBD tools — those show you the block hierarchy; MDK's STEP export shows you the physical assembly derived from the same model.

### Digital twin + real-time feedback

The CDKd concept at its full realisation: a running physical system with MDK-connected telemetry that continuously compares measured timeseries against the simulation prediction, flags divergence as a maintenance signal, and proposes model updates when divergence exceeds a threshold. Not "the infrastructure is configured correctly" (AWS Config does that) but "the physical system is behaving as the physics model predicted." 

For a water resource engineer: is the Yarrangobilly catchment evapotranspiring at the rate the Odum ESL model predicts, or has something changed (vegetation removal, drought, soil compaction)? The MDK digital twin tells you, continuously, without a modeller having to manually recalibrate.

### NEAT-evolved models as published constructs

If NEAT discovers a model topology that fits an observed timeseries, that topology is a publishable `@mdk/provider-*` construct. The community could build a library of empirically validated model structures — not from datasheets, but from real-world timeseries calibration. This is closer to how biological systems are understood: you observe the behaviour, fit a compartmental model, and the fitted structure is the hypothesis.

A community NEAT-evolved construct library — with fitness scores, validation datasets, and provenance — would be genuinely novel in the systems modelling world.

## Monetisation Strategies

### Honest assessment of the market

Open-source physical simulation frameworks exist (OpenModelica, SciPy). What they lack: code-first CDK-style composability, cloud-native integration, version-controlled models, the CDKd feedback loop, and a commercial component catalog. MDK's differentiation is the combination, not any individual element.

### Strategy 1: Open core

Core framework (BG/Odum kernel, TypeScript constructs, CLI, validation) is MIT licensed and free. Revenue from:
- **Hosted CDKd validation service** — the "how it started vs how it is going" loop running in the cloud. Subscription per fleet. High stickiness once connected.
- **Enterprise team collaboration** — shared model registry, team-level construct library, compliance report generation.
- **Verified component catalog** — vendors pay an annual listing fee for verified, team-maintained `@mdk/provider-<vendor>` packages. Not the community packages — those are free on npm. The "MDK Verified" badge on a Daikin HVAC construct is worth something to the vendor.

### Strategy 2: Domain-specific SaaS

Pick one vertical (water resource management is the natural first), build a hosted MDK application that:
- Ingests Hydstra data automatically
- Runs the calibration harness on your catchment model
- Generates compliance reports (EPBC Act, state water sharing plans, WSUD)
- Alerts when measured inflows diverge from model predictions

Government agencies and environmental consultancies pay for this kind of tool. The underlying MDK framework stays open; the SaaS layer charges for hosting, support, and domain-specific integrations.

### Strategy 3: Consulting and implementation

Build MDK stacks for clients. Each engagement validates the framework on a new domain (HVAC, water, industrial), produces publishable case studies, and feeds improvements back to the open-source project. Funds development without requiring community traction first.

### Strategy 4: Academic licensing and grants

The Giannantoni IDC implementation, the NEAT model evolution capability, and the bond graph validation engine are genuinely novel research tools. ARDC (Australian Research Data Commons), ARC (Australian Research Council), and EU Horizon grants are realistic funding paths for the research infrastructure components. This is not primarily a revenue strategy but it funds R&D and establishes academic credibility.

## What Should Actually Happen Next

In priority order:

1. **Implement `ModelConstruct` / `ModelStack`** as CDK-style base classes in `@mdk/core`. The design is done (`examples/construct-pattern/`). Remove `@ts-nocheck` as each class is implemented.

2. **Implement `@mdk/provider-odum`** — `Store`, `Source`, `Sink`, `Flow`, `Site`, `SiteRegistry` as real TypeScript classes with working Hydstra API integration. This turns the design sketches into runnable code.

3. **Add parameter calibration harness** (T0.8) — extend `advanced.c` with CMA-ES optimizer. This plus `Site.fromHydstraApi()` gives you the complete backtesting loop for one real catchment.

4. **Build the GitHub Pages demo** (T5.4) — the interactive WASM demo is still the highest-ROI unbuilt thing for user acquisition. No marketing beats "open this URL, draw a model, see it simulate."

5. **Add CDKd validation Lambda** (T4.4) — the "how it started vs how it is going" loop. Once the simulation output is stable and a real IoT deployment is connected via `@mdk/provider-aws`, implement the validator Lambda. This is when MDK becomes defensibly different from everything else.

---

## Bottom Line

The 2026-04-27 review said "subtraction, not addition." This session added things — but they were additive in the right way. The provider model is cleaner than the domain model. The CDK-style construct API is the right abstraction that was missing. The CDKd loop is the commercial moat. The component catalog with BOM generation is the practical value proposition.

The risk is still scope. The building exhibition excited the right commercial instinct but the right response is "water resource management first, HVAC second, everything else after that." Not "HVAC, hydraulics, pharma, finance, and CAD simultaneously."

The core is sound and getting sounder. The discipline needed is to finish `@mdk/provider-odum` on one real catchment before writing another design sketch.
