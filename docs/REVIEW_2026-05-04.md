## V-Model Fidelity Assessment

The V-Model (`docs/V_MODEL.md`) defines a left-side decomposition path, a vertex, and a right-side verification path. Here is how each node maps to current implementation:

### Left Side — Decomposition

| V-Model Step | Tool / Class | Status |
|---|---|---|
| User Requirements → System Architecture | `create_project` (MCP) | **Implemented** — takes plain-English description, decomposes to blocks with SQLite persistence |
| System Architecture → Logical Design | `create_block` (MCP) | **Implemented** — hierarchical blocks with parent FK, domain field |
| Logical Design → Physical Implementation | `refine_block` (MCP) | **Implemented** — LLM→BG generation + optional NEAT evolutionary refinement |
| SysML v2 structural subset | `create_model` with `domain: 'sysml'` | **Implemented** — Gemini structured output → SysmlPackage JSON |
| SysML → Bond Graph transpilation | `transpile_sysml` (MCP) | **Implemented** — PartUsage/PortUsage/FlowConnectionUsage → BondGraphModel |

The left side is substantially present. The critical gap here is the **Socratic loop** described in `CONTEXT_ASSEMBLY.md` Phase 1.2: the system currently has no mechanism to halt generation and ask the user clarifying questions when boundary conditions are missing. `create_model` will guess rather than ask. This gap matters because it is where physical hallucinations enter — an LLM that estimates `k` coefficients rather than asking for them produces a model that validates formally but is physically meaningless.

### Vertex — Physical Implementation

| V-Model Step | Tool / Class | Status |
|---|---|---|
| Bond Graph model (WASM kernel) | `@mdk/sim-kernel` C/WASM | **Implemented** — Euler/RK4, SCAP, state-space |
| NEAT structural evolution | `packages/core/src/evolution/` | **Implemented** — genome, population, speciation, requirement fitness |
| Flat-graph constraint | `flattenBondGraph` | **Not implemented** — WASM still flat-only; BG sub-system composition is design-only |

### Right Side — Verification

| V-Model Step | Tool / Class | Status |
|---|---|---|
| Component validation (SCAP) | `validate_model` (MCP) | **Implemented** — multi-stage: JSON parse → Zod → WASM SCAP causality |
| Sub-system simulation (TDC/IDC) | `run_simulation` (MCP) | **Implemented** — both TDC (WASM RK4) and IDC (log-Euler) paths wired |
| System verification (Emergy/AHP/LCC) | `compute_emergy` (MCP) | **Partially implemented** — emergy/transformity present; AHP and LCC are absent |
| Requirements traceability | `verify_requirements` (MCP) | **Implemented** — SysML RequirementUsage checked against simulation output |
| BOM generation | `generate_bom` (MCP) | **Implemented** — structured BOM JSON from model; commercial metadata schema exists but no live distributor API |

### Traceability Links (Dashed in V-Model)

| Link | Status |
|---|---|
| REQ → VERI (validation) | `verify_requirements` partially covers this |
| SYS → SUBS (verification) | Indirect — `assemble_model` → `run_simulation` path works |
| LOG → UNIT (consistency) | `validate_model` at the block level covers this |

**Overall V-Model verdict:** The skeleton is present. Every node in the diagram has at least a stub. The right side is stronger than the left — validation/simulation is more complete than the elicitation/decomposition path. The weakest point is the gap between "user says something" and "system has a physically valid boundary condition" — the Socratic loop and formal parameter elicitation are absent.

## New Subsystems: What Was Built

### SQLite State Store (`packages/mcp-server/src/store/db.ts`)

This resolves the statefulness gap identified in memory. The schema is correct: `projects → blocks → relations → generation_log`. Foreign key integrity, WAL mode, UUIDs as PKs. `MDK_DB_PATH` override for isolated test environments. One gap: **there is no `validation_results` table** as was planned — validation JSON is stored as a column on `blocks`, which works for simple cases but collapses the multi-stage validation trace into a single blob.

### IDC Solver (`packages/core/src/solvers/idc-solver.ts`)

The Giannantoni log-Euler scheme is correctly implemented: `x(t+h) = x(t) · exp(h · f/x)` with a zero-guard fallback to standard Euler. Reference is cited. The `IdcStateSpace` type is compatible with WASM A/B matrix output. **Gap:** the drift metric — the comparison between IDC and TDC trajectories that Giannantoni analyses — is not computed. `run_simulation` exposes `calculus: 'tdc' | 'idc'` but there is no `calculus: 'both'` mode that returns the divergence Δx(t) between the two solvers. That divergence is the scientifically interesting output; without it, IDC is available but its value as a probe of generative causality is latent.

### NEAT Evolution (`packages/core/src/evolution/`)

Four files: `neat.ts` (genome, innovation tracking, speciation), `population.ts` (generation loop), `fitness.ts` (generic fitness interface), `requirement-fitness.ts` (RequirementUsage → fitness). The innovation number scheme is correct (NEAT-paper-compatible). `refine_block` wires this as `evolutionary: true` with a `generations` parameter. **Gap:** the `PopulationManager` does not yet implement crossover — only mutation (add/remove node, add bond, toggle bond). Speciation via compatibility distance is in `NeatGenome` but population-level speciation pools are not used by `PopulationManager`. This means the system does NEAT mutation without NEAT speciation, which degrades topological diversity. Fine for an MVP, but should be labelled as a partial NEAT implementation.

### Emergy / Transformity (`packages/core/src/emergy/transformity.ts`)

Present and wired to `compute_emergy`. No issues identified with the scope of what is there.

## Architectural Gaps

### Gap 1: The CDK-Style Construct Layer Is Still Missing

`ModelConstruct` and `ModelStack` (T2.6, T2.7) were identified as the highest-priority unbuilt thing in the April-30 review. They are still absent. The existing API (`MdkApp`, `MdkStack`, `MdkSystem`) works but does not enforce the discipline of "Construct = pure physics, Stack = synthesis boundary." The `examples/construct-pattern/` sketches remain aspirational TypeScript with `@ts-nocheck`. Until this layer is built, the CDK analogy in the documentation is misleading — users who come from CDK will not find the patterns they expect.

### Gap 2: `@mdk/provider-odum` Does Not Exist

`Store`, `Source`, `Sink`, `Flow`, `Site`, `SiteRegistry` are referenced throughout the documentation but no implementation exists. This is the most critical gap for the primary target user (water resource engineers, ecologists). Every claim about Hydstra integration, backtesting, and Site-level calibration rests on this package. Until it is real code, those claims are marketing copy.

### Gap 3: No Recursion Gate

`CONTEXT_ASSEMBLY.md` Phase 4.4 describes the recursion gate: when a block is tagged as a subsystem, the engine spawns a new Phase 1 instance for it. This is the mechanism that makes the V-Model hierarchical rather than flat. It is not implemented. `refine_block` currently treats every block as a leaf — it generates a flat BG model for whatever block it is asked to refine, without detecting that the block should itself be decomposed further. The `complexity_tag` field described in the DSEE spec has no counterpart in the SQLite schema.

### Gap 4: The DSEE Pipeline Is a Spec, Not Code

`docs/CONTEXT_ASSEMBLY.md` defines a 14-step pipeline (7 LLM + 7 deterministic). The MCP tools cover Steps 4.2 (Bond Graph synthesis, SCAP), 4.3 (state-space), and a portion of 1.3 (requirements). Steps 1.1 (Use Case extraction), 1.2 (Socratic loop), 1.3 (formal requirement flow-down), 2.2 (BDD complexity tagging), 2.3 (IBD port routing), 3.1–3.3 (behavioral synthesis), and 4.1 (traceability audit) are either absent or are handled implicitly by the LLM in `create_model`. The distinction between "LLM-driven" and "deterministic" steps exists in the spec but not in the code.

### Gap 5: BG Sub-System Composition Is Flat

The WASM kernel only handles flat graphs. The `BgSubSystem` schema, `flattenBondGraph()`, and `composeBondGraphs()` planned in memory (project_hierarchical_generation.md) are not implemented. This means multi-block models cannot currently be simulated as a unit — `assemble_model` concatenates block JSONs but the assembled model has no port bonds connecting them. Any multi-block simulation is physically incorrect until sub-system port composition is implemented.

### Gap 6: `@mdk/sim-kernel` Is Not Published on npm

T0.1 has an open checkbox: "Publish as `@mdk/sim-kernel` on npm with pre-compiled WASM binary." The WASM binary must be committed or pre-built into the package for consumers. Currently the project requires the Apple Container toolchain to build from source. This means nobody external can `npm install @mdk/sim-kernel` and use it.

### Gap 7: GitHub Pages Demo Does Not Exist

T5.4 ("GitHub Pages HTML/WASM interactive demo") is the highest-ROI unbuilt user acquisition tool. The April-30 review said this clearly. It is still not built. Every other user-facing gap is an implementation problem; this one is a market problem — there is no way for someone to discover MDK and immediately see it work without cloning the repo and running `make`.

## MCP Security: The Local MCP Problem

You raised a specific security concern about running MCP servers locally. The current `.mcp.json`:

```json
{
  "mcpServers": {
    "mdk": {
      "command": "node",
      "args": ["packages/mcp-server/dist/index.js"],
      "env": {
        "ANTHROPIC_API_KEY": "${ANTHROPIC_API_KEY}"
      }
    }
  }
}
```

This is exactly the pattern described as high-risk. Here is the MDK-specific risk analysis:

### What the Current Setup Does

Every time Claude Code (or any MCP client) starts, it launches `node packages/mcp-server/dist/index.js` on your machine. This process:
- Runs with your user permissions
- Has access to `ANTHROPIC_API_KEY` and `GEMINI_API_KEY` in environment
- Reads and writes `.mdk/state.db` in the current working directory
- Makes outbound HTTPS calls to `generativelanguage.googleapis.com`
- Has `packages/mcp-server/src/tools/create-model.ts` executing arbitrary LLM-generated JSON injected back into the process

The supply chain attack vector: `packages/mcp-server` has npm dependencies (`@modelcontextprotocol/sdk`, `zod`, `better-sqlite3`). If any transitive dependency is compromised (as happened with `axios`, `litellm`, `bitwarden` cited in your briefing), the next `npm install` or restart injects the compromised code into a process that holds your API keys and has write access to your model database.

### MDK-Specific Mitigations Needed

MDK is in an unusual position here: it is building an MCP server *and* consuming it locally. The server is MDK's own code, which reduces the supply chain attack surface compared to installing a third-party MCP. But the dependencies are still third-party, and the execution context is still your machine.

The Lambda handler stub (`packages/mcp-server/src/lambda.ts`) is the correct long-term architecture — T10.2 moves the server to AWS Lambda where:
- (a) It runs in an isolated sandbox, not on your machine
- (b) API keys are managed by IAM/Secrets Manager, not environment variables on your laptop
- (c) Invocation logs go to CloudWatch (audit)
- (d) IAM resource policies on the Lambda provide tool-level access control

**Recommended short-term mitigations while T10.2 is not built:**

1. **Dependency pinning** — pin all `packages/mcp-server/package.json` dependencies to exact versions (`"zod": "3.23.8"` not `"^3.23.8"`). This prevents a compromised package version from being pulled in on restart. Add `npm audit` to `make test`.

2. **API key isolation** — do not put `ANTHROPIC_API_KEY` in `.mcp.json`'s `env` block. The MDK MCP server itself does not need the Anthropic key (it uses Gemini). Remove it. `GEMINI_API_KEY` should be the only key in scope.

3. **Read-only SQLite for the MCP server** — the current `db.ts` creates and writes to `.mdk/state.db`. Consider whether the MCP server needs write access at all for the common read-heavy use case, and add a `MDK_DB_READONLY=1` environment flag.

4. **HTTP transport for monitoring** — the `--transport http` mode already exists. Running the server as a persistent HTTP process rather than stdio-spawned allows you to add request logging and rate limiting at the HTTP layer. This is point (c) from the briefing ("monitor and audit tool calls/responses").

5. **Tool schema as a deny-list gate** — the MCP SDK allows tool registration to be conditional. Consider gating destructive tools (`refine_block` with `evolutionary: true`, `create_project`) behind an environment flag (`MDK_ALLOW_WRITES=1`) so read-only tool calls cannot trigger LLM synthesis or DB writes when the flag is absent.

**The real answer is T10.2.** The Lambda deployment moves MDK's threat model from "trusted local process" to "sandboxed remote service." That is architecturally the right place for this server to run. The Lambda stub needs to be completed. Until then, the mitigations above reduce but do not eliminate the risk.

## DSEE Pipeline vs. V-Model: Are They Consistent?

`docs/CONTEXT_ASSEMBLY.md` (DSEE) and `docs/V_MODEL.md` describe the same system from different angles. They are mostly consistent but there is a naming and framing tension:

- The V-Model is requirements-centric (the V is a verification traceability diagram).
- The DSEE spec is process-centric (it is a numbered pipeline with LLM/deterministic annotations).

The V-Model maps tool names directly onto nodes. The DSEE names steps (1.1 Context Extraction, 2.2 BDD, 4.2 SCAP) without mapping them to MCP tools. This means a user looking at either document alone will not see the full picture. **The V-Model should reference the DSEE step numbers, and the DSEE should reference the MCP tool names.** Currently they are two documents describing the same system in parallel without cross-referencing.

The DSEE's "balance is exactly 50/50: 7 LLM steps + 7 deterministic steps" is the right intellectual frame. The V-Model implies the same thing but does not make it explicit. This is the MDK conceptual core: LLM for semantic translation, WASM for physics proof. Both documents should say this in the same words.

## What Should Actually Happen Next

Ordered by impact, not by ambition:

1. **Fix the MCP security posture** (days, not weeks). Pin dependencies, remove the `ANTHROPIC_API_KEY` from `.mcp.json`, add `npm audit` to CI. This is low effort and reduces real risk now.

2. **Implement `flattenBondGraph()` and BG port composition** (T0 hierarchy work). Without this, multi-block assembly produces invalid physics. The IDC solver and NEAT evolution are running on single-block flat graphs. The V-Model bottom vertex — where blocks from the decomposition path meet — is currently broken for any system with more than one block.

3. **Build the GitHub Pages demo** (T5.4). One well-composed demo page with the RC circuit running live in the browser is worth more for user acquisition than everything else on this list combined. The WASM binary exists. The browser demo is blocked only by the time it takes to write the HTML wrapper.

4. **Implement `ModelConstruct` / `ModelStack`** (T2.6, T2.7). Remove `@ts-nocheck` from `examples/construct-pattern/`. This is the API surface that actual users will write. Until it is real code, there is no onboarding path for a TypeScript engineer who does not already know Bond Graph theory.

5. **Build a minimal `@mdk/provider-odum`** with `Store`, `Source`, `Sink`, `Flow` — without Hydstra integration but with working `mdk synth()`. This turns the ecological/hydrological domain from a design sketch into a working package. Hydstra can be added later. The CDK-style API without Hydstra is already useful.

6. **Add the IDC drift metric** to `run_simulation`. If `calculus: 'both'` is selected, return both TDC and IDC trajectories and compute the elementwise Δx(t). This is one additional function call in `run-simulation.ts` and turns IDC from "available" to "scientifically useful."

7. **Complete T10.2 (Lambda handler)** to migrate off local MCP. This is the structural fix for the MCP security issue. The stub is already in place.

## Bottom Line

The system has more depth than it did a week ago: IDC, NEAT, emergy, SQLite state, 14 MCP tools, `verify_requirements`, `transpile_sysml`. That is genuine progress on the physics and agentic layer.

The V-Model is implemented in skeleton form. Every node has a representation. The right side (verification) is stronger than the left side (elicitation). The vertex (BG flat simulation) works for single-block models but breaks for multi-block assemblies. The traceability links are partial.

The two things that would move the needle most for users — the GitHub Pages demo and the CDK-style construct API — are still not built, for the third review in a row. The system keeps building deeper into the physics stack. That is intellectually satisfying and technically sound. But at some point the question is not "is the physics correct?" (it is) but "can anyone use this?" (not yet).

The MCP security issue is real and worth addressing now with the short-term mitigations. The long-term answer (T10.2 Lambda) is architecturally correct and already partially in place.
