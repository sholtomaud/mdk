# MDK Concept Review
**Date:** 2026-04-27  
**Reviewer:** Claude Sonnet 4.6

---

## Verdict

The core insight is sound and genuinely novel. The roadmap is dangerously overloaded and will kill the project if not cut hard.

---

## What Is Actually Good

**The kernel idea is strong.** Bond Graphs as an AST for physical systems, compiled by a C/WASM engine, surfaced through a TypeScript CDK-style API — that is a real insight, not just a clever analogy. Bond Graph causality is a deterministic algorithm, which makes real-time linting genuinely tractable. Phase 1 proves it works.

**Version control for engineering artefacts is a real pain point.** The industry argument is solid. Engineers absolutely do lose design decisions in proprietary binaries. A Git-diffable physical model has real value.

**The VSCode DX angle (Phase 5) is the right wedge.** Real-time physical causality errors as red squiggles is a compelling, demonstrable feature. It is the thing that would make an engineer say "oh" and care.

---

## Honest Problems

### 1. The roadmap is nine phases but should be three

Phases 1–2–5 form a coherent, shippable v1: working C engine, TypeScript constructs, VSCode linting. That is a product. Everything from Phase 6 onward is a research programme disguised as a feature list. Trying to build all nine in parallel will result in none of them being finished.

### 2. Phase 9 (Cyber-Physical Cloud Link) has a modelling flaw at its foundation

Treating AWS Lambda P99 latency as a deterministic `e^{-sT}` delay block is an undergraduate-level simplification that will produce wrong answers in production. Cloud latency is stochastic with heavy tails. P99 is not P50. Bursty traffic changes the distribution entirely. Building a "compile-time safety check" on top of a deterministic model of inherently non-deterministic infrastructure will give engineers false confidence — which is worse than no check at all. The concept is intellectually exciting but needs to be framed honestly as a probabilistic tool, not a formal guarantee.

### 3. Phase 7 (Formal Verification) is multiple PhDs, not a product phase

KeYmaera X and SpaceEx are active research tools. Integrating them for arbitrary user-defined ODEs, with safety envelope proofs, for a general mechatronic framework — that is a serious academic research problem. Listing it as a sprint task misrepresents the difficulty. If formal verification is genuinely important to the vision, it deserves a separate research track with an academic partner, not a checkbox list.

### 4. Phase 8 (Automated DFMEA) is mislabelled

What is described is automated sensitivity analysis and fault injection — which is valuable. It is not DFMEA. Real DFMEA requires: curated failure mode databases, occurrence ratings based on field data, detection ratings based on inspection methods, and domain expert judgment on severity. The automated version catches parameter extremes; it misses failure modes that require domain knowledge to even enumerate (contamination, manufacturing variation, wear degradation curves). Overselling it as "DFMEA" will lose the credibility of engineers who know the difference.

### 5. The hardware registry is a chicken-and-egg problem with no proposed solution

The registry has value only if vendors publish packages. Vendors publish only if there are users. There are users only if the registry has packages. This is a standard two-sided market bootstrapping problem, and the current TASKS.md has no strategy for it beyond "build the tooling." A realistic plan would start with the team hand-curating 20–30 high-value components (Maxon motors, common Arduino shields, Raspberry Pi GPIO) before launch.

### 6. `mdk research` (T3.3) is a liability risk in safety-adjacent software

An LLM inferring Bond Graph parameters (R, L, k_t, J, B) from an Amazon product listing will be wrong in ways that are hard to detect. Motor parameters require either lab measurement or a detailed engineering datasheet — not a marketing description. A silently wrong parameter in a safety-relevant model is more dangerous than no model. This feature should be removed from the roadmap or explicitly gated behind a manual verification step with prominent warnings.

### 7. The target user is very small

The intersection of "engineers who write TypeScript," "engineers who know Bond Graph theory," and "engineers building mechatronic systems" is currently tiny. Bond Graph theory is taught in a handful of graduate programmes. The framework assumes the user already knows the physics modelling paradigm. The roadmap has no phase for onboarding, education, or expanding that user base.

---

## What Should Happen

**Cut to a v1 scope:** Phase 1 (done), Phase 2 (TypeScript constructs + CLI), Phase 5 (VSCode extension + real-time linting). Ship that. It is a complete, useful, demonstrable product.

**Reframe Phase 9** as a research direction with an explicit disclaimer that the cloud-latency-as-physics model is approximate and probabilistic. Make the uncertainty visible in the output, not hidden.

**Rename Phase 8** to "Automated Sensitivity Analysis & Fault Simulation." Drop the DFMEA label until the failure mode database exists.

**Park Phase 7** entirely. Put a note in the docs that formal verification is a long-term research goal. Do not build infrastructure for it until there is a clear academic or industrial partner.

**Solve the registry cold-start problem** before building Phase 3 tooling. Curate the first 20 packages manually. Tooling without content is not useful.

---

## Bottom Line

The core of this project — Bond Graphs as code, WASM causality linting, TypeScript CDK pattern — is a real contribution. The risk is that the vision document has grown to encompass every possible adjacent idea, and the TASKS.md now reflects that. A project that attempts to simultaneously solve Bond Graph modelling, formal verification, fleet SRE safety analysis, automated DFMEA, and LLM-powered hardware discovery will ship none of them. The discipline needed here is subtraction, not addition.
